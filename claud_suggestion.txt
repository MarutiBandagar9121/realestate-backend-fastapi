Property Node Tree Architecture
Commercial Real Estate Platform — Architecture Proposal

1. Executive Summary
This document proposes the core data architecture for the Commercial Real Estate Platform, specifically addressing the most complex design challenge: how to model real estate properties that vary dramatically in physical structure — from large multi-building campuses with wings, to simple malls, to open land parcels.
The proposed solution is the Property Node Tree — a flexible, unified data model that handles all property structures without duplication or rigid schema constraints.

2. The Core Problem
Commercial real estate properties do not follow a single fixed structure. Our platform must handle all of these simultaneously:
Property TypeStructureOffice CampusProperty → Building → Wing → Floor → UnitOffice BuildingProperty → Building → Floor → UnitMall / RetailProperty → Floor → UnitWarehouseProperty → Building → UnitCoworking CentreProperty → Building → Floor → ProductsLand ParcelProperty onlyStandalone ShopProperty → Unit
A naive approach — separate tables for buildings, wings, floors, units — forces nullable foreign keys everywhere and breaks under edge cases.

3. The Solution: Property Node Tree
Core Idea: Every level of a property's physical hierarchy — building, wing, floor, unit — is simply a "node." Each node knows its type and its parent. This is identical in concept to a folder structure on a computer.
My Computer          →   Property (e.g. Oberoi Commerz)
  └── Documents      →     └── Building A       (node: building)
        └── Work     →           └── East Wing  (node: wing)
              └── file →               └── Floor 9 (node: floor)
                                              └── Unit 9B (node: unit)
All of this lives in one table called property_nodes. Each row has a node_type (building / wing / floor / unit) and a parent_node_id pointing to another row in the same table.

4. The Two Core Tables
Table: properties
Holds master information — name, address, city, micromarket, GPS, asset category, total area, ownership type (own vs third-party operator).
Table: property_nodes
Holds every building, wing, floor, and unit as individual rows. Key columns:

node_type — building / wing / floor / unit
parent_node_id — points to parent row in same table (NULL for top-level)
property_id — which property this node belongs to
area_sqft, floor_number, sequence_order, status
owner_id — units can have independent owners
attributes (JSONB) — vertical-specific details like cabin count, dock height etc.


5. Why This Handles All Property Types
Office Campus with Wings:
Building A → East Wing → Floor 9 → Unit 9A, Unit 9B
Mall (no wings):
Floor 1 → Shop G-01, Shop G-02. Wing level simply does not exist. No nulls, no empty rows.
Land Parcel:
Zero node rows. The property record itself is the asset.
Warehouse:
Building → Unit (no floors needed). Only the levels that exist in reality are created.
Each property gets exactly the structure it needs — nothing more, nothing less.

6. Commercial Offerings (Lease / Sale)
The physical structure and the commercial transaction are kept fully separate. A unit exists physically. What you do commercially with it is stored in a separate offerings table.
This means the same unit can simultaneously have:

A lease offering at ₹120 psf/month
A sale offering at ₹18,000 psf

Both are just two rows in the offerings table pointing to the same unit node. No duplication of physical data.
Importantly, an offering can be attached to any node level — not just units. A whole building or an entire floor can also have an offering, which is common in large enterprise deals.

7. Coworking as a Special Case
Coworking spaces follow the same node tree for physical structure, but instead of traditional unit-level offerings, they have coworking_products — hot desks, dedicated desks, private cabins, meeting rooms — each with their own inventory counts and pricing. This sits alongside the main offerings table without polluting it.

8. Database Choice: PostgreSQL
We recommend PostgreSQL over MySQL for three specific reasons relevant to this design:
Recursive tree queries — fetching an entire property tree (for stacking plans) requires recursive queries. PostgreSQL handles these significantly better and faster than MySQL.
JSONB indexing — vertical-specific attributes (cabin count, dock height, floor load etc.) are stored as JSONB. PostgreSQL's JSONB can be indexed and filtered efficiently. MySQL's JSON support cannot.
PostGIS — future location-based queries ("find properties within 2km of BKC") require geospatial support. PostgreSQL + PostGIS is the industry standard.
Google Cloud SQL fully supports PostgreSQL, so infrastructure does not change.

9. Key Design Principles
Separation of concerns — physical structure, commercial offerings, and vertical-specific attributes are three separate concerns stored in three separate places.
No rigid hierarchy — the system enforces valid node structures per property type through a config table, not hardcoded schema constraints. Adding a new property type requires adding one config row, not altering tables.
Extensibility — new asset types, new transaction types, new verticals can be added without schema redesign.
Ownership at unit level — different units within the same building can have different owners, all tracked cleanly.

10. What Requires Stakeholder Decision
Before engineering begins, the following need sign-off:

PostgreSQL confirmed as the primary database (replacing MySQL)
JSONB for vertical attributes accepted as the attribute storage strategy
Node tree approach approved as the property hierarchy model
Confirmation that any node level (building / floor / unit) can carry a commercial offering
